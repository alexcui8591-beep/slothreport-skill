# 报告骨架蓝图（提炼自 DAPO 范本）

> 这是默认结构。按论文类型可裁剪，但「前置先行」和「逐改动 diff 表」两条不能动。

## 标准章节

### 一、简介（Why care）
- 一句话说清这个方法解决什么问题、属于什么范式。
- **核心贡献 bullet 列表**：每条 = 英文术语 + 一句中文效果。范本：
  - **Clip-Higher** — 促进多样性，避免熵坍缩
  - **Dynamic Sampling** — 提高训练效率与稳定性
  - **Token-Level Policy Gradient Loss** — long-CoT RL 场景关键
  - **Overlong Reward Shaping** — 降低奖励噪声、稳定训练
- 目标：读者 30 秒知道「这篇为什么值得读」。

### 二、前置回顾（Build the foundation）—— **不可省**
找到主角的 baseline / 灵感来源，**先把它讲透**：
- **机制概述**：一段话讲清 baseline 在干什么（GRPO：采样 G 个回答→算优势→奖高罚低→多轮优化 θ）。
- **损失函数**：写出 baseline 的 loss，**逐符号拆解**（见 style_devices 的「公式拆解」）。
- **一个跑通的例子**：用具体输入端到端走一遍（范本用 `2+3*4=?`，4 列表格：生成回复→算奖励→模型迭代→持续迭代）。
- 目标：读者拿着 baseline 的脑模型，才能 diff 出主角的改动。

### 三、主体浅析（The main act）
- **先放主角的损失函数 / 核心公式**，紧跟一张 **baseline↔主角 diff 总表**（`维度 | baseline | new | Reason`），把所有改动点一次性列清。
- 然后**逐个改动单独成节**，每节用四段式：
  1. **问题 / 现象**：先讲不改会怎样（熵坍缩现象、零梯度浪费、长序列梯度稀释、截断噪声）。
  2. **根源**：为什么会这样（对称 ε 裁剪对低概率 token 太严）。
  3. **解法**：怎么改（只提上界不动下界），配公式 + 具体数字推演。
  4. **效果**：改完好在哪，配实验图 / 曲线。
- 每节内部尽量再放一张小对比表或一个数字例子。

### 四、实验（Land it）—— **一等章节，不可压缩**，详见 `references/experiments_section.md`
解读的可信度全在这。六块齐全、图文并茂（结果图保留原图 + 每图配解读）：
- **Setup**：基座、超参精确（优化器、lr、batch、G、ε_low/high、温度、topP）、评测集与指标。
- **主结果**：保留原文主结果图，给 baseline→本方法的提升与含义。
- **消融（重头戏）**：逐技术贡献，点明每个是"提分/增稳/提效"。读者最想知道"提升靠哪个"。
- **训练动态**：长度/奖励/熵/生成概率，讲清为什么重要、怎么读异常、过拟合信号。
- **关键现象图**：支撑各改动的证据图（熵坍缩曲线、收敛更快图…），每张配"它证明了什么"。
- **一句话结论**：实验整体说明了什么。

### 五、总结（Takeaway）
- 一段话收束：这个方法对 baseline 做了哪几处「精妙洞察与修改」，解决了什么。
- **必附 Paper Link** + 原文标题图。

## 章节顺序的内在逻辑
简介(钩子) → 前置(地基) → 主体(逐 diff 深挖) → 实验(证据) → 总结(收束)。
**地基没打牢就讲主体 = 读者掉队。** 这是范本最关键的编排决策。
